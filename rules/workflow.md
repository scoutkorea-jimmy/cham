# 작업 절차

이 저장소는 **여러 사람과 여러 AI 도구가 함께** 수정합니다.
그래서 "무엇을 만들 것인가"보다 **"지금 코드가 어떤 상태인가"를 먼저 확인하는 절차**가 더 중요합니다.

---

## 1. 최우선 — 편집 전 외부 변경 감지

이전 대화, 이전 작업, 마지막 분석 이후로 코드가 바뀌었을 수 있습니다.
**이전 이해가 아직 유효하다고 가정하지 않습니다.**

기존 파일을 수정하기 전에 현재 코드를 확인하고, 알고 있던 구조·동작과 비교합니다.
다른 사람이나 도구가 추가·삭제·재작성·이동·이름변경·구조변경한 흔적이 보이면:

1. **해당 코드 편집을 멈춥니다.**
2. 무엇이 바뀌었는지 사용자에게 알립니다.
3. 영향받는 파일·함수·컴포넌트·라우트·스키마·의존성을 특정합니다.
4. 그 변경이 왜 중요한지, 현재 요청과 어떻게 충돌하는지 설명합니다.
5. 구체적인 통합 선택지를 제시하고 **확인을 받은 뒤** 진행합니다.

```text
외부 변경 감지

영향 파일:
- path/to/file

무엇이 바뀌었나:
- 감지된 변경의 명확한 설명

왜 중요한가:
- 아키텍처 · 동작 · 데이터 흐름 · 요청 작업에 미치는 영향

가능한 조치:
1. 새 구현을 유지하고 요청 작업을 거기에 맞춘다
2. 이전 구현을 복원한다
3. 두 구현의 일부를 병합한다
4. 충돌 코드를 먼저 검토한 뒤 결정한다

어느 쪽으로 진행할지 확인해 주세요.
```

다른 사람의 변경을 말없이 덮어쓰거나, 되돌리거나, 재해석하거나, 버리지 않습니다.
새 코드가 옳다고도, 내 이전 버전이 옳다고도 가정하지 않습니다.

> 서식·주석·생성 메타데이터처럼 무해한 차이로는 멈추지 않습니다.
> **동작 · 아키텍처 · 데이터 · 의존성 · 요청한 구현에 영향을 줄 때만** 멈춥니다.

이 규칙은 "즉시 진행하라", "질문 최소화하라", "확인 없이 완료하라"는 모든 지시보다 우선합니다.

---

## 2. 편집 전 조사

코드를 쓰거나 고치기 전에 현재 구현을 확인합니다. 확인 대상:

기능 진입점 · 관련 라우트와 페이지 · import된 컴포넌트 · 공용 컴포넌트 · 상태 소유자 ·
API 호출 위치 · DB 접근 위치 · 검증 로직 · 비즈니스 규칙 · 공용 타입 · 유틸 함수 ·
환경변수 · 설정 파일 · 관련 테스트 · **유사한 기존 구현**

새 함수·컴포넌트·서비스·훅·유틸·타입·API 클라이언트를 만들기 전에
**같은 역할을 하는 기존 구현을 먼저 찾습니다.**

파일 이름에만 의존하지 않습니다. 동작·import 심볼·엔드포인트 이름·UI 문구·데이터 필드·
도메인 용어로 검색합니다.

## 3. 저장소 상태 확인

매 작업 시작 시 저장소를 **낯설고 바뀌었을 수 있는 코드베이스**로 취급합니다.
파일 변경·추가·이동·이름변경, 의존성·라우트·스키마·환경변수 변경,
중복 구현 발생 여부, 이미 요청을 해결하는 구현의 존재 여부를 확인합니다.

바뀐 코드베이스에 옛 계획을 그대로 적용하지 않습니다.
파일을 열어볼 수 있을 때 기억으로 코드를 재생성하지 않습니다.

> 저장소가 현재의 사실이지만, **그 안의 모든 구현이 옳다고 가정하지는 않습니다.**

---

## 4. 변경 계획 (사소하지 않은 변경 전)

```text
현재 구조:
- 관련 기존 구현

근본 원인:
- 문제의 실제 원인

수정할 파일:
- 정확한 경로

생성할 파일:
- 정확한 경로와 필요한 이유

재사용할 코드:
- 기존 함수 · 컴포넌트 · 타입 · 서비스

제거·통합할 코드:
- 중복 · 폐기 · 충돌 코드

영향 범위:
- 영향받을 수 있는 기존 동작

검증:
- 수행할 타입체크 · 린트 · 테스트 · 빌드 · 수동 확인
```

여러 파일에 걸친 큰 변경은 이 계획 없이 시작하지 않습니다.
작고 고립되고 위험이 낮은 수정은 긴 설명 없이 진행해도 되지만, **관련 코드는 반드시 먼저 확인**합니다.

---

## 5. 변경 범위 통제

요청한 작업에 필요한 것만 고칩니다. 다음은 요청 없이 하지 않습니다:

무관한 페이지 재디자인 · 대규모 이름변경 · 프레임워크 교체 · 상태관리 교체 ·
DB 구조 변경 · 라우트 재구성 · 전체 포맷팅 · 무관한 패키지 업그레이드 · 무관한 모듈 리팩터

더 큰 변경이 정말 필요하면 **실행 전에 이유를 설명**합니다.

> 최소 변경은 "가장 작은 패치"가 아니라 **"가장 작은, 완결되고 구조적으로 올바른 변경"** 입니다.

### 전체 파일 덮어쓰기 금지

일부만 고치면 될 때 파일 전체를 다시 쓰지 않습니다. 전체 재작성은 다른 사람의 작업을 지우고,
주석과 예외 처리를 날리고, 리뷰를 어렵게 만듭니다.

---

## 6. 컨텍스트 · 토큰 통제

매 작업마다 저장소 전체를 읽지 않습니다. 다음 순서로 좁힙니다:

1. 요청한 기능이나 오류를 검색
2. 담당 라우트·페이지·컴포넌트·서비스를 특정
3. 그 import와 직접 의존성을 추적
4. 관련 타입·스키마·테스트 확인
5. 필요할 때만 범위를 넓힘

`node_modules` · `.next` · `dist` · `build` · 캐시 · 커버리지 · 바이너리 · 미디어 ·
대용량 로그 · DB 덤프 · 번들 · (의존성 해석이 필요한 경우가 아니면) 락파일은 열지 않습니다.

같은 작업 중 이미 본 파일을 (바뀌지 않았다면) 다시 읽지 않고, 같은 분석을 반복 서술하지 않습니다.

---

## 7. 검증

구현 후 이 프로젝트에서 가능하고 관련 있는 검증을 모두 수행합니다.
이 사이트는 정적 사이트라 **실제 브라우저 구동이 주 검증 수단**입니다 —
절차는 `.claude/skills/verify` 와 [responsive.md](responsive.md) 4절을 따릅니다.

확인 대상: 요청한 동작 · 관련 기존 동작 · 빈 상태 · 잘못된 입력 · 로딩 · 오류 ·
권한 · 모바일 레이아웃 · 데스크톱 레이아웃 · 접근성

**하지 않은 테스트를 했다고 하지 않습니다.**
검증할 수 없었다면 무엇을·왜 못 했고 사용자가 어떻게 확인하면 되는지 정확히 밝힙니다.

## 8. 완료 보고

```text
변경 목적
- 해결한 문제

수정한 파일
- path/to/file: 정확한 변경 내용

생성한 파일
- path/to/file: 생성 이유

삭제한 파일
- path/to/file: 삭제 이유

재사용한 기존 코드
- 컴포넌트 · 함수 · 타입 · 서비스

구조적 결정
- 이 구현을 선택한 이유

처리한 외부 변경
- 보존 · 병합 · 제외한 타인의 변경

수행한 검증
- 타입체크 / 린트 / 테스트 / 빌드 / 수동 확인

남은 위험 · 후속 작업
- 실제로 남아 있는 항목만
```

보고는 사실만 적습니다. 수행하지 않은 검증을 적지 않습니다.

---

## 9. 중단 조건 — 진행 전에 물어봐야 하는 상황

- 다른 사람의 변경이 요청 작업과 충돌할 때
- 같은 기능을 하는 구현이 둘 존재하고, 어느 쪽이 살아 있는지 불명확할 때
- 데이터 마이그레이션이 필요하거나 데이터 손실 가능성이 있을 때
- 인증·인가 동작이 바뀔 수 있을 때
- 공개 API 계약이 깨질 수 있을 때
- 주요 의존성 교체나 광범위한 구조 재작성이 필요할 때
- 기존 동작 보존이 제품 결정 없이는 불가능할 때

멈출 때는 **구체적인 충돌과 선택지**를 제시합니다.
"어떻게 할까요?" 같은 막연한 질문 대신, 결정 가능한 질문을 합니다.

> 예: "현재 브랜치에 참가자 검증 흐름이 둘 있습니다. `services/registration.ts` 의 새 흐름은
> 불완전 레코드를 거부하고, `pages/register.tsx` 의 기존 흐름은 임시저장을 허용합니다.
> 임시저장을 유지할까요, 전역 엄격 검증으로 통일할까요, 최종 제출 시에만 검증하도록 병합할까요?"

---

## 10. 매 작업 실행 순서

1. 저장소 현재 상태 확인
2. 이전 이해 이후의 변경 감지
3. 관련된 외부 변경이 있으면 **중단하고 보고**
4. 기존 구현 검색
5. 데이터·제어 흐름 추적
6. 근본 원인 또는 확장 지점 특정
7. 변경 계획 제시
8. 필요한 코드만 수정
9. 이번 변경으로 쓸모없어진 코드 제거
10. 검증 수행
11. 최종 diff에서 무관한 변경 확인
12. 완료 보고

**요청이 단순해 보인다는 이유로 단계를 건너뛰지 않습니다.**
여러 사람과 AI 도구가 동시에 손대는 저장소에서는 단순한 요청도 코드베이스를 망가뜨릴 수 있습니다.
