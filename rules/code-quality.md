# 코드 품질 규칙

핵심 원칙: **요청한 기능이 동작하게 만드는 것으로 끝내지 않는다.**
구조적 복잡도 · 중복 · 모호함 · 유지보수 비용을 늘리지 않으면서 변경한다.

기존 구현이 구조적으로 나쁘면 그대로 확장하지 않고, 요청 기능을 안전하게 추가하기 위한
**최소한의 리팩터**를 먼저 합니다.

---

## 1. 스파게티를 키우지 않는다

책임이 뒤엉키고, 조건이 반복되고, 로직이 중복되고, 데이터 흐름이 불분명한 코드에
**같은 구조로 로직을 더 얹지 않습니다.**

> "기존 코드가 원래 이렇게 되어 있어서 같은 패턴을 따랐습니다" — 이렇게 말하지 않습니다.
> 나쁜 패턴은 이미 존재한다는 이유로 정당해지지 않습니다.

스파게티를 건드려야 할 때:
1. 구체적인 구조 문제를 특정한다
2. 최소한의 안전한 리팩터 경계를 정한다
3. 직접 영향받는 영역만 리팩터한다
4. 기존 동작을 보존한다
5. 정리된 구조를 통해 요청 기능을 추가한다
6. 이번 변경으로 쓸모없어진 코드를 제거한다
7. 영향받는 모든 경로를 검증한다

요청하지 않은 전체 재작성은 하지 않습니다.

## 2. 책임을 섞지 않는다

하나의 파일·컴포넌트·함수가 렌더링 · 상태관리 · API 요청 · DB 접근 · 데이터 변환 ·
비즈니스 규칙 · 검증 · 인가 · 영속화 · 분석 · 오류 보고를 **불필요하게 함께** 갖지 않습니다.

독립적으로 재사용·테스트·변경 가능한 것들은 분리합니다.
다만 구조적 이유 없이 잘게 쪼개지 않습니다 — 목표는 **명확한 책임**이지 파일 개수가 아닙니다.

## 3. 함수

함수 하나는 하나의 명확한 목적을 갖습니다. 다음이면 리팩터합니다:

무관한 작업을 함께 수행 · 깊게 중첩된 조건 · 여러 독립 상태를 변경 ·
일관되지 않은 반환 형태 · 숨은 외부 상태에 크게 의존 · 다른 함수의 로직 중복 ·
**이름을 명확히 지을 수 없음**

의미 없는 이름을 쓰지 않습니다: `handleData` `processAll` `doSomething` `runLogic`
`tempFix` `newHandler` `finalFunction` `fixIssue` `helper2`

실제 책임을 설명하는 이름을 씁니다: `validateParticipantRegistration` `fetchEventAttendance`
`calculateCheckInCount` `normalizeParticipantRecord` `authorizeEventManager`

불명확한 이름이나 구조를 **주석으로 때우지 않습니다.**

## 4. 컴포넌트

컴포넌트를 만들기 전에 같은 책임의 기존 컴포넌트를 찾습니다. 적절하면 확장하거나 조합합니다.
기존 것을 고치기 번거롭다는 이유로 거의 같은 컴포넌트를 새로 만들지 않습니다.

한 페이지 컴포넌트 안에 API 호출 · 데이터 변환 · 폼 검증 · 비즈니스 규칙 · 로딩 상태 ·
권한 · 전체 UI 렌더링을 모두 넣지 않습니다.

프로젝트에 이미 확립된 UI 패턴이 있으면 두 번째 패턴을 도입하지 않습니다
(→ [components.md](components.md)).

## 5. 중복은 결함이다

새 로직을 구현하기 전에 프로젝트 전체에서 유사 로직을 검색합니다.
API 호출 · 검증 · 포맷팅 · 오류 처리 · 권한 확인 · 상태 전이 · 데이터 변환 ·
상수 · 타입 · 스키마 · UI 컴포넌트.

의미 있는 중복이 있으면 재사용하거나 통합합니다. **코드를 복사해 몇 줄만 바꾸지 않습니다.**

반대로, 실제 중복이 생기기 전에 일반화된 추상을 미리 만들지 않습니다.
줄 수를 줄이려고 무관한 비즈니스 규칙을 하나의 과도하게 설정 가능한 함수로 합치지 않습니다.

## 6. 증상을 패치하지 않는다

보고된 한 사례를 통과시키려고 조건 예외를 추가하지 않습니다.
버그를 고치기 전에 기대 동작 · 실제 동작 · **최초로 갈라지는 지점** · 원인이 되는 상태나
데이터 변환 · 근본 원인을 특정합니다.

금지되는 증상 패치:
- 잘못된 상태를 고치지 않은 채 null 체크만 반복 추가
- 특정 사용자·식별자·라우트·날짜·값 하드코딩
- 깨진 로직을 우회하는 불리언 추가
- 실제 흐름을 우회하는 대체 경로 신설
- 예외 삼키기 / 무한 재시도 / 크래시 방지용 빈 객체 반환

## 7. 타입 안전성

타입 오류를 없애려고 `any` 를 쓰지 않습니다. 컴파일을 통과시키려고 타입을 약화하지 않습니다.

선호: 명시적 도메인 타입 · 공용 인터페이스 · 구별 유니온 · `unknown` + 타입 가드 ·
스키마 검증 · 정확한 반환 타입

같은 데이터 구조를 여러 파일에서 따로 정의하지 않습니다 — 공용 타입은 **단일 원본**을 씁니다.
`any` 가 정말 불가피하면 범위를 제한하고, 왜 필요한지 / 어디에 쓰는지 / 언제 제거할지 설명합니다.

## 8. 데이터 변환의 일관성

같은 API 응답을 여러 컴포넌트에서 다르게 변환하지 않습니다.
외부 데이터는 일관된 레이어에서 정규화합니다.

원본 API 데이터 · 정규화된 도메인 데이터 · UI 표시용 데이터의 경계를 흐리지 않습니다.
공용 데이터 형태를 바꿀 때는 **모든 소비처를 먼저 확인**합니다.

## 9. API · 서비스

프로젝트에 서비스/클라이언트 레이어가 있으면 UI 컴포넌트에 직접 API 호출을 흩뿌리지 않습니다.
새 요청을 만들기 전에 기존 API 클라이언트 · 엔드포인트 래퍼 · 공용 인증 처리 ·
오류 정규화 · 캐싱 · 재시도 · 응답 타입을 확인합니다.

같은 엔드포인트를 서로 다른 구현으로 여러 번 호출하지 않습니다.
URL · 시크릿 · 토큰 · 계정 ID · 환경별 값을 하드코딩하지 않습니다.

## 10. 상태 관리

**단일 원본을 중복시키지 않습니다.** 새 상태를 도입하기 전에 소유자 · 읽는 곳 · 바꾸는 곳 ·
성격(로컬/공유/서버/캐시/영속) · 이미 그것을 표현하는 소스가 있는지 확인합니다.

같은 상태의 사본 둘을 이펙트로 동기화하지 않습니다(불가피한 경우 제외).
기존 상태에서 안전하게 파생할 수 있는 값에 상태를 추가하지 않습니다.
한 컴포넌트의 데이터에 전역 상태를 쓰지 않고, 앱 전체 일관성이 필요한 데이터에 로컬 상태를 쓰지 않습니다.

## 11. 오류 처리

**오류를 숨기지 않습니다.** 아래는 금지:

```ts
try { await executeTask(); } catch { /* 무시 */ }
// @ts-ignore
// eslint-disable-next-line
const result = response || {};
```

원인을 해결하거나 문서화하지 않은 채 타입·린트·런타임 오류를 억제하지 않습니다.

사용자 대상 오류 · 검증 오류 · 인가 오류 · 네트워크 오류 · 복구 가능 오류 ·
개발자 대상 실패를 구분합니다. 민감 정보 노출 없이 유용한 맥락을 제공합니다.

## 12. 의존성

패키지를 설치하기 전에 확인합니다 — 동등한 의존성이 이미 있는지 · 프레임워크가 제공하는지 ·
작은 로컬 구현으로 충분한지 · 활발히 관리되는지 · 호환되는지 · 번들/보안 비용이 정당한지.

사소한 유틸을 위해 큰 라이브러리를 넣지 않습니다.
추가할 때는 **왜 필요한지 · 어디에 쓸지 · 기존 의존성으로 왜 부족한지 · 어떤 대안을 검토했는지**
밝힙니다. 무관한 의존성을 함께 업데이트하지 않습니다.

## 13. 쓸모없어진 코드 제거

새 구현이 기존 코드를 대체하면 더 이상 쓰이지 않는 것을 제거합니다 —
미사용 import · 함수 · 컴포넌트 · 타입 · 스타일 · API 래퍼 · 폐기된 라우트 ·
임시 로그 · 주석 처리된 레거시 · 백업 파일 · 미사용 패키지.

**추측으로 삭제하지 않습니다.** 정적·동적 참조를 모두 검색한 뒤 지웁니다.
동적 import · 라우트 규약 · 설정 기반 로딩 · 리플렉션은 특히 조심합니다.

## 14. 버전 접미사 파일 금지

`new` `old` `final` `final2` `copy` `backup` `v2` `latest` `fixed` `temp` 를
파일명이나 심볼명에 쓰지 않습니다.

금지 예: `DashboardNew.tsx` · `authServiceV2.ts` · `finalHandler.ts` · `UserCardCopy.tsx` · `api-fixed.ts`

대체 구현이면 올바른 구현을 갱신하고, 참조를 확인한 뒤 낡은 쪽을 제거합니다.
두 버전이 제품상 이유로 공존해야 하면 **시간 순서가 아니라 실제 역할로** 이름 짓습니다.

## 15. 주석은 '왜'를 설명한다

코드를 그대로 반복하는 주석을 쓰지 않습니다.

```ts
// 나쁨
// 사용자 이름을 설정한다
setUserName(name);
```

주석은 다음을 설명할 때만 씁니다 — 자명하지 않은 설계 결정 · 외부 시스템의 제약 ·
비즈니스 규칙 · 호환성 요구 · 보안 고려사항 · 더 단순해 보이는 구현이 왜 위험한지.

임시 코드에는 이유와 제거 조건을 명시합니다. 막연한 TODO를 남기지 않습니다.
